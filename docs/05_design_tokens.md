# 디자인 토큰

이 문서는 UI 디자인 토큰 자체보다, Gaesup-State demo와 도구 화면에서 지켜야 할 UI 기준을 정리합니다.

Gaesup-State는 개발자 도구 성격이 강합니다. 따라서 화면은 화려한 랜딩 페이지보다 상태, 검증 결과, 성능 수치를 빠르게 읽을 수 있는 쪽이 좋습니다.

## 기본 방향

- 정보 밀도를 적당히 높입니다.
- 카드 안에 또 카드를 넣지 않습니다.
- 상태 배지는 짧고 명확하게 씁니다.
- 성공, 격리, 차단 상태가 한눈에 구분되어야 합니다.
- 수치와 계약 정보는 표 형태를 우선합니다.

## 상태 색상

| 상태 | 의미 |
| --- | --- |
| Runs shared | host 계약과 맞아 공유 실행 |
| Runs packaged | bundled dependency로 패키지 내부 실행 |
| Runs isolated | 공유 store 대신 격리 store 사용 |
| Blocked | 실행 차단 |

색상은 보조 정보일 뿐입니다. 반드시 텍스트로도 상태를 표시해야 합니다.

## dependency isolation 화면

각 패키지 카드는 다음 정보를 가져야 합니다.

- 패키지 이름과 버전
- 실행 결정
- dependency source
- store contract
- error 또는 warning code
- 사람이 이해할 수 있는 설명

## counter demo 화면

counter demo는 복잡한 설명보다 동작 확인이 중요합니다.

- 네 프레임워크 카드가 같은 count를 표시해야 합니다.
- 버튼을 누르면 모든 카드가 같이 갱신되어야 합니다.
- shared count와 각 카드 count가 어긋나면 문제로 봅니다.

## 문서 스타일

- 문서는 한국어를 기본으로 합니다.
- 이모지는 사용하지 않습니다.
- 깨진 인코딩 문자는 남기지 않습니다.
- API 이름, 파일명, 명령어는 코드 스타일로 표시합니다.
- 메인 README는 처음 보는 사람도 이해할 수 있게 씁니다.
- 세부 문서는 구현자가 바로 참고할 수 있게 구체적으로 씁니다.
